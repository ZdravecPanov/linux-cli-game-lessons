Shared Directory Lockout

This lesson explains why a user can be denied access to a directory that looks writable, how to prove the failing layer, and what a safe shared-directory fix looks like.

What Is Broken

The user is in the correct group and the target directory may even look group-writable, but one parent directory blocks traversal. On Linux, you do not just need write permission on the final directory. You also need execute permission on every parent directory in the path.

How It Works

For directories, permission bits mean something different than for files:

  • r means you can list names
  • w means you can create, rename, or remove entries
  • x means you can traverse into the directory

What To Prove

  • Which user is failing
  • Which groups that user belongs to
  • Which parent path element blocks traversal
  • Whether the final target is actually the real problem

How To Investigate

Use identity checks and path-aware inspection, not only a quick ls on the target:

id
namei -l /tmp/linux-game/shared/ops
ls -ld /tmp/linux-game/shared /tmp/linux-game/shared/ops

namei -l is especially useful because it shows every directory in the path and its permissions.

Common Mistakes

  • Jumping straight to chmod 777
  • Checking only the target directory
  • Assuming group membership alone is enough
  • Ignoring traversal on parent directories

Good Operational Pattern

For shared admin folders, prefer a dedicated group, controlled group-write, and setgid on the shared directory. Fix the minimum layer that restores the intended access model. Do not widen permissions more than the scenario requires.

What Success Looks Like

  • The affected user can create a file in the target directory
  • The directory is not made world-writable
  • You can explain exactly which parent permission caused the failure
