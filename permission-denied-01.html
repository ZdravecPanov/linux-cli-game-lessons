Shared Directory Lockout

This lesson is a small practical guide to Linux directory permissions. The goal is to understand why a directory can look writable and still fail, how parent traversal works, and how an administrator proves the failing layer before changing permissions.

Situation

A teammate cannot create a file inside a shared operations directory. Your job is to prove whether the real failure comes from ownership, group membership, directory traversal, or write permission before you make any change.

Mission Goal

  • Confirm which user is failing
  • Inspect the full path, not only the final target directory
  • Find the exact permission layer blocking access
  • Restore safe shared-write behavior without opening permissions too broadly

What Layer This Is

This mission lives in the filesystem access-control layer. The application is not the problem here. The real problem is how Linux evaluates identity, groups, directory traversal, and write permissions across the full path.

What We Are Using and Why

The main tools are id, namei -l, and ls -ld.

  • id tells you who the current user really is and which groups they belong to
  • namei -l shows every element in a path with ownership and permissions
  • ls -ld lets you inspect specific directories directly

An administrator uses these tools together because access failures often come from the path around the target, not only from the target itself.

How It Works in Real Operations

For directories, Linux permission bits mean something very specific:

  • r lets you list names
  • w lets you create, remove, or rename entries
  • x lets you traverse into the directory

A user can fail to create a file even if the final directory looks writable, because Linux also requires execute permission on every parent directory in the path.

How Linux Evaluates Directory Access

When Linux decides whether a user can create a file in a directory, it does not ask only one question. It checks several layers in order.

  • First: who is the user right now
  • Second: is that user the owner, a member of the directory's group, or only part of "others"
  • Third: can that identity traverse every parent directory in the path
  • Fourth: does the final directory allow the needed write and execute access

This is why shared-directory problems often confuse people. They look at the final target, see group write, and assume everything should work. But Linux may already have rejected the request one directory earlier because the user cannot traverse into that parent path element.

Why This Fails in Real Life

This kind of issue is common in teams because shared directories are usually built in layers. One admin creates the top-level path, another creates the working directory, and later someone changes permissions quickly during an incident.

  • The user is in the correct group, but one parent directory is missing execute permission
  • The target directory itself is fine, but the path to it is not
  • A broad fix like chmod 777 makes the symptom disappear, but destroys the intended security model

A good administrator learns to treat Permission denied as a path problem, an identity problem, or a final-target problem, then prove which one it is.

How To Use It in Practice

A strong admin workflow here is:

  • Confirm the exact user identity first
  • Inspect the full directory chain, not only the target
  • Compare group ownership and execute bits on every parent path element
  • Prove whether the final target is truly the failing layer
  • Repair only the permission model needed for the intended shared access

What matters is the reasoning: identify the actor, inspect the path, isolate the failing layer, then fix only that layer.

Choosing the Safest Fix

In real administration work, the best fix is usually the smallest fix that restores the intended access model.

  • If the wrong user owns the path, ownership may be the issue
  • If the user is missing from the intended shared group, group membership may be the issue
  • If the user is in the right group but cannot traverse the path, mode bits on a parent directory may be the issue
  • If a shared directory should keep new files under one common group, setgid on that directory is often part of the right design

The point is not to memorize one magic command. The point is to match the fix to the proven layer. That is what separates a safe admin fix from a risky one.

What Good Evidence Looks Like

  • The user identity and group membership are confirmed before the fix
  • The full path shows exactly where traversal breaks
  • The final directory is not changed blindly before path evidence is clear
  • The user can create the file after the minimal safe fix

What Not To Do

  • Do not jump straight to chmod 777
  • Do not inspect only the target directory
  • Do not assume group membership alone guarantees access

Common Problems Administrators See

  • The target is group-writable, but a parent directory blocks traversal
  • The right group is present, but execute permission is missing on a parent path
  • The fix widens permissions too far and breaks the shared-access model

What You Should Learn From This Mission

The real lesson is to ask a better operational question. Not only "what are the permissions on the target?" but "can this exact identity traverse every parent and write at the final layer?"
