Broken Nginx Site Config

This lesson is a practical guide to config-driven web service troubleshooting. The goal is to understand what Nginx is responsible for, why config validation comes first, and how to separate syntax, listener state, and HTTP behavior.

Situation

An internal site is not serving traffic correctly. You need to determine whether the problem begins in Nginx configuration, then prove the service listens and returns the expected page after the fix.

Mission Goal

  • Confirm whether the problem is in config syntax or not
  • Repair the actual Nginx config issue
  • Prove the service is listening on the expected port
  • Verify the site returns the expected response

What Layer This Is

This mission lives in the web-service and configuration layer. The key question is not only whether the process exists, but whether the service can parse its config, listen correctly, and serve the expected content.

What We Are Using and Why

Nginx is a web server and reverse proxy. The main tools here are nginx -t, ss, and curl.

  • nginx -t validates config syntax
  • ss shows whether a listener exists on the expected port
  • curl verifies the actual HTTP behavior, not just process state

How It Works in Real Operations

For config-driven services, syntax errors break everything upstream of runtime verification. If the config is invalid, there is no point guessing about firewalls, upstream apps, or browser behavior yet.

A strong troubleshooting order is:

  1. Validate the config
  2. Fix the exact error reported
  3. Confirm the service is listening
  4. Verify the endpoint over HTTP

How Nginx Thinks About a Request

Nginx has to survive several layers before a user ever sees a working page.

  • First: the configuration must parse successfully
  • Second: the service must start and bind to the expected address and port
  • Third: a matching server block must answer the request
  • Fourth: the requested content must exist and be returned correctly

That is why good troubleshooting here is layered. A syntax error is different from a listener problem, and a listener problem is different from a wrong HTTP response.

How To Use It in Practice

A strong admin workflow here is:

  • Start with syntax validation instead of random edits
  • Read the exact file and line number of the reported error
  • Inspect the config that is actually loaded
  • Check whether the service is listening only after validation succeeds
  • Run a final HTTP test so you prove the service is usable

What matters is the reasoning: syntax first, listener second, endpoint third.

Choosing the Safest Fix

The safest repair is the smallest change that matches the exact failure you proved.

  • If nginx -t points to one syntax problem, fix that exact directive first
  • If syntax is valid but the port is missing, investigate service state or binding next
  • If the port is open but the page is wrong, move up to HTTP behavior rather than re-editing syntax blindly

A strong admin does not mix all layers together. They restore one broken layer at a time and verify after each step.

What Good Evidence Looks Like

  • The config validates cleanly before you claim success
  • The service is listening on the expected port
  • The endpoint returns the expected body, not just a TCP listener
  • The fix matches the exact syntax or config problem found

What Not To Do

  • Do not start by blaming networking before syntax validation
  • Do not edit random files before reading the exact error
  • Do not stop after the listener appears without testing HTTP

Common Problems Administrators See

  • A missing semicolon or invalid directive blocks startup
  • The process starts, but does not listen on the expected port
  • The port is open, but the actual response is still wrong

What You Should Learn From This Mission

The real lesson is to separate "config valid", "service listening", and "endpoint healthy". They sound similar under pressure, but they are not the same thing operationally.
