Broken Nginx Site Config

This lesson covers the right troubleshooting order for a config-driven web service: validate syntax, inspect the exact file and line, confirm listening state, and verify the endpoint over HTTP.

What Is Broken

Nginx has an active site config that contains a syntax error. Until the config validates, the service cannot start cleanly and there is no point guessing about ports, firewalls, or upstream behavior.

How Nginx Failure Usually Shows Up

  • nginx -t fails with a file and line number
  • No process is listening on the expected port
  • curl to the endpoint fails or returns nothing

The config test is usually your highest-signal first step.

How To Investigate

nginx -t
cat /etc/nginx/conf.d/linux-game.conf
ss -lntp | grep 8080
curl http://127.0.0.1:8080/

This sequence keeps the troubleshooting layered: syntax, config content, listener state, then actual HTTP response.

Common Mistakes

  • Editing random config files before reading the error line
  • Trying a restart before proving the config is valid
  • Confusing "process started" with "service is healthy"
  • Stopping after the listener appears without testing with curl

Good Operational Pattern

For config-driven services, think in this order:

  1. Validate the config
  2. Fix the exact error reported
  3. Retest syntax
  4. Start or reload the service
  5. Verify the actual endpoint

What Success Looks Like

  • nginx -t succeeds
  • Nginx listens on 8080
  • curl http://127.0.0.1:8080/ returns the expected page body
