Internal Name Resolution Failure

This lesson is a practical guide to separating service health from name resolution. The goal is to understand why an app can work by IP and still fail by hostname, and how an administrator narrows the failure to the resolver layer.

Situation

An internal application is healthy when reached locally, but a teammate says the hostname does not work. Your task is to prove the application layer is fine, isolate the resolver problem, and restore the expected internal name.

Mission Goal

  • Prove the application is healthy by IP first
  • Confirm the hostname is the failing layer
  • Restore local name resolution cleanly
  • Verify the same endpoint works by hostname

What Layer This Is

This mission lives in the networking and resolver layer. The main question is whether the problem is in the application itself, the port, the listener, or the translation from a hostname to an address.

What We Are Using and Why

The main tools are ss, curl, and getent hosts.

  • ss shows whether something is listening
  • curl proves whether the endpoint works over HTTP
  • getent hosts uses the system resolver path, which makes it very useful for real host-resolution checks

How It Works in Real Operations

Hostname failure is not the same as service failure. If an app works on 127.0.0.1 but fails by name, the service may already be healthy. That means the smarter move is to narrow the problem to resolution, not to rebuild or reconfigure the app blindly.

How Name Resolution Is Evaluated

When you reach a service by hostname, Linux does more than open a TCP connection. It first has to translate the name into an address through its resolver path.

  • First: the system checks the configured name-resolution sources
  • Second: the hostname must resolve to the expected address
  • Third: only then can the client connect to the correct listener

This is why comparing hostname behavior with IP behavior is so powerful. If IP works but the hostname fails, you have already removed a big part of the application stack from suspicion.

How To Use It in Practice

A strong admin workflow here is:

  • Confirm a listener exists on the expected port
  • Test the endpoint directly by IP or localhost first
  • Check whether the hostname resolves through the system resolver
  • Compare the successful IP-based check with the failing hostname-based check
  • Fix only the resolution layer, then retest the hostname

What matters is the reasoning: first prove the app, then prove the name is the problem.

Choosing the Safest Fix

The safest fix is to restore the expected name-to-address mapping without touching the application unnecessarily.

  • If the service is already healthy by IP, do not edit the app
  • If the hostname does not resolve, repair the local resolution layer
  • If it resolves to the wrong address, correct the mapping instead of changing the port or service

The goal is to fix the narrowest failing layer. Good admins resist the urge to rebuild healthy layers just because the symptom appears at the client side.

What Good Evidence Looks Like

  • The application responds by IP before the resolver is changed
  • The hostname fails while the IP-based endpoint succeeds
  • The hostname resolves correctly after the fix
  • The same health check works by hostname at the end

What Not To Do

  • Do not change the app before proving it already works by IP
  • Do not call it a DNS issue without comparing IP and hostname behavior
  • Do not stop after fixing resolution without rerunning the hostname check

Common Problems Administrators See

  • The app is healthy, but the internal hostname does not resolve
  • The hostname resolves, but points to the wrong address
  • The wrong layer gets changed because the operator never proved the app by IP first

What You Should Learn From This Mission

The real lesson is to reduce the fault domain quickly. If localhost works and the hostname fails, the service and port are already mostly exonerated. That is strong troubleshooting discipline.
