Internal Name Resolution Failure

This lesson teaches a layered troubleshooting approach for hostname problems: prove the app works by IP, then isolate resolver failure before changing anything else.

What Is Broken

The service itself is healthy on localhost, but the internal hostname does not resolve. This is not an app outage. It is a name-resolution problem.

How To Think About It

When a hostname fails, split the problem into layers:

  1. Is something listening?
  2. Does the endpoint work by IP?
  3. Does the name resolve?
  4. Does the same request work by hostname after the fix?

Useful Commands

ss -lntp | grep 8088
curl http://127.0.0.1:8088/health
getent hosts inventory.internal
curl http://inventory.internal:8088/health

getent hosts is especially useful because it uses the system resolver path instead of bypassing it.

Common Mistakes

  • Changing the app when localhost already works
  • Assuming "DNS issue" without proving the app is healthy by IP
  • Skipping a final hostname-based curl after the fix

Good Operational Pattern

Always narrow the fault domain first. If IP works and the hostname fails, the service and port are already mostly exonerated. Fix the smallest resolver layer that restores the expected name.

What Success Looks Like

  • The service responds on 127.0.0.1:8088
  • inventory.internal resolves locally
  • The hostname-based health check works end to end
