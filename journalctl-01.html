Find Failure in the Journal

This lesson is a practical guide to reading service evidence from the journal. The goal is to understand why systemctl status and journalctl answer different questions, and how to use both to isolate a startup blocker quickly.

Situation

A metrics forwarder stopped after a cleanup action. The service manager confirms failure, but the real reason is hidden in the journal. You need to find the startup blocker, repair it, and verify the unit is healthy again.

Mission Goal

  • Confirm the service is failing before changing anything
  • Use journalctl to find the specific startup error
  • Repair the missing dependency and prove the service is healthy again

What Layer This Is

This mission lives in the service-observability layer. The service manager can tell you that a unit failed, but the journal often tells you why it failed and which runtime dependency actually broke.

What We Are Using and Why

The main tools are systemctl status, journalctl, filesystem inspection, and a post-fix health check.

  • systemctl status shows the unit state and high-level failure context
  • journalctl -u ... shows the unit's recent log lines in sequence
  • filesystem inspection helps confirm whether the missing path or dependency is real
  • the health check proves the service recovered after the repair

How It Works in Real Operations

Many service failures are not obvious from status alone. The unit may simply show "failed" while the real story is a missing directory, bad dependency, permission issue, or invalid runtime path buried in the service's own output.

That is why experienced admins read both state and evidence: status narrows the unit, journal narrows the cause.

How the Journal Changes the Investigation

The journal is where a generic failure often becomes a concrete one.

  • Status may say the unit failed
  • The journal may say a directory is missing, a path is unreadable, or a dependency could not be opened

That difference is critical. Without the journal, you are guessing. With the journal, you can often point to one missing runtime dependency and ignore ten unrelated possibilities.

How To Use It in Practice

A strong admin workflow here is:

  • Confirm the service is really failed
  • Read the most recent journal lines for the unit
  • Translate the error into a concrete filesystem or runtime check
  • Repair only the dependency the logs point to
  • Restart and verify both service state and health output

What matters is the reasoning: let the journal narrow the repair target before you change anything.

Choosing the Safest Fix

The safest fix is to repair the exact blocker the journal describes, not to rewrite the whole service environment.

  • If the logs point to a missing directory, prove it and restore only that directory
  • If they point to permissions, verify the path and ownership before changing modes broadly
  • After the repair, recheck both the unit state and the service health path

This keeps the repair small, explainable, and easy to verify.

What Good Evidence Looks Like

  • The unit state is confirmed before edits begin
  • The journal points to one concrete startup blocker
  • The missing path or dependency is verified on disk
  • The service is healthy only after that blocker is repaired

What Not To Do

  • Do not restart repeatedly without reading the logs
  • Do not assume the unit file itself is wrong before checking runtime evidence
  • Do not stop after restart without verifying a healthy state

Common Problems Administrators See

  • The unit state is visible, but the real blocker is a missing directory or path
  • The operator reads only status and misses the key journal line
  • The service starts after a fix, but nobody verifies the health path

What You Should Learn From This Mission

The real lesson is that status answers "what state is the unit in?" while the journal often answers "what exactly broke?" Good troubleshooting needs both.
