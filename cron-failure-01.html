Cron Job Fails on Schedule

This lesson is a practical guide to one of the most common Linux automation traps: a command works when you run it manually, but fails from cron. The goal is to understand why scheduled jobs have different execution conditions and how to prove the environment is the real failure layer.

Situation

An internal report-digest task is scheduled to run automatically, but its scheduled run keeps failing. The worker script exists, yet the cron execution path is still broken. Your job is to identify why the schedule fails and restore a healthy scheduled run.

Mission Goal

  • Inspect the cron definition and the cron output first
  • Prove the worker exists and the failure belongs to the scheduled environment
  • Fix the cron job safely without rewriting the script logic
  • Verify the scheduled run succeeds afterward

What Layer This Is

This mission lives in the automation and scheduled-operations layer. The core question is not "does the script exist?" but "under what environment does cron actually run it?"

What We Are Using and Why

The main tools are the cron definition, the cron output log, filesystem inspection, and a scheduled-run simulation.

  • the cron file shows the real scheduled command and environment
  • the cron log shows what happened when the scheduled run fired
  • filesystem inspection proves whether the worker exists and is executable
  • rerunning the scheduled path proves the fix works in cron conditions, not only in your shell

How Scheduled Jobs Differ From Your Shell

Cron jobs usually run with a much smaller environment than an interactive shell. They may not load your shell profile, your aliases, or the same command search path.

  • A command may work interactively because your shell knows where to find it
  • The same command may fail from cron because cron cannot resolve that command name
  • This often shows up as "command not found" even though the script exists on disk

That is why scheduled-task troubleshooting is often really environment troubleshooting.

How To Use It in Practice

A strong admin workflow here is:

  • Read the cron definition first
  • Read the latest cron output before editing anything
  • Confirm the worker exists and is executable
  • Compare what cron can see with what your interactive shell can see
  • Fix the scheduled job definition and rerun the scheduled execution path

What matters is the reasoning: prove the script is healthy, then prove cron is running it under broken conditions.

Choosing the Safest Fix

The safest repair is the one that makes scheduled execution explicit.

  • If cron cannot find the command, make the path explicit in the cron layer
  • If the script itself is healthy, do not rewrite it just to hide a scheduling problem
  • After the fix, rerun the scheduled path instead of only running the worker manually

Good admins fix the job definition or execution environment first, because that is the layer that actually failed.

What Good Evidence Looks Like

  • The cron file is inspected before editing
  • The log shows a scheduled-environment failure, not a missing script on disk
  • The worker script is verified as present and executable
  • The scheduled run succeeds only after the cron definition is corrected

What Not To Do

  • Do not assume cron has the same PATH as your shell
  • Do not edit the worker script before proving the job definition is broken
  • Do not stop after a manual script run succeeds

Common Problems Administrators See

  • The script exists, but cron cannot find it by command name
  • The scheduled environment is missing required variables or a usable PATH
  • The operator tests only interactively and misses the real scheduled failure path

What You Should Learn From This Mission

The real lesson is that scheduled-job incidents often come from execution context, not from script logic. When a cron job fails, ask what cron actually knew at runtime.
