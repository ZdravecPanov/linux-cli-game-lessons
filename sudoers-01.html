Broken Limited Sudo Access

This lesson is a practical guide to least-privilege sudo delegation. The goal is to understand how sudoers rules match users and commands exactly, why limited access can fail even when the intended command exists, and how to repair the rule without giving away broad admin power.

Situation

An account named auditor is supposed to run one approved maintenance command with sudo, but the delegation is broken. Your job is to find the exact rule failure, repair it safely, and prove the user still has only narrow access.

Mission Goal

  • Prove the affected user cannot run the approved delegated command
  • Inspect the sudoers drop-in that owns the access rule
  • Fix the rule without granting broad sudo access
  • Validate the sudoers file and verify the limited command works

What Layer This Is

This mission lives in the privilege-delegation layer. The important question is not "can someone become root somehow?" but "is the exact intended command delegated safely and correctly?"

What We Are Using and Why

The main tools are sudo -l, sudoers drop-ins, filesystem inspection, and visudo.

  • sudo -l -U <user> helps show what a user is allowed to run
  • the sudoers drop-in contains the exact command delegation
  • filesystem inspection confirms the real command path on disk
  • visudo -cf validates the sudoers file safely

How Sudoers Matching Really Works

Sudo delegation is exact, not approximate.

  • The user must match
  • The host context must match
  • The absolute command path must match

If the sudoers file allows the wrong path, the user can still be blocked even though the intended command exists on disk. That is why comparing the delegated path with the real command path is such a strong troubleshooting move.

How To Use It in Practice

A strong admin workflow here is:

  • Test the problem from the affected user's point of view
  • Inspect the drop-in rule that owns the delegation
  • Compare the allowed command path with the real path on disk
  • Validate the file before and after the fix
  • Confirm the approved command works while broad sudo remains blocked

What matters is the reasoning: repair the exact delegation layer, not the user's role in the whole system.

Choosing the Safest Fix

The safest fix is the narrowest fix that restores the intended command and nothing more.

  • If the rule points to the wrong command path, repair the path
  • If the file is invalid, validate and correct the sudoers syntax
  • Do not grant full sudo access when only one command needs delegation

Strong administrators do not widen authority just because a least-privilege rule is temporarily broken.

What Good Evidence Looks Like

  • The failure is reproduced as the affected user
  • The drop-in rule is tied to the real delegated command path
  • The sudoers file validates cleanly after the fix
  • The approved command works, while broad sudo remains blocked

What Not To Do

  • Do not add the user to general sudo access just to "make it work"
  • Do not edit sudoers without validation
  • Do not stop after the approved command works unless you also verify least privilege is preserved

Common Problems Administrators See

  • The rule allows the wrong absolute command path
  • The intended command exists, but sudoers references a typo or old path
  • The access is "fixed" by granting far too much privilege

What You Should Learn From This Mission

The real lesson is that sudo delegation is a precision tool. When limited access breaks, the right answer is usually to repair the exact rule, not to broaden privilege.
