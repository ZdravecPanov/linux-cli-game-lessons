Log-Driven App Recovery

This lesson focuses on using logs as the shortest path from a vague symptom to a precise bad config value.

What Is Broken

A startup-check script reads config from an environment file. One value is invalid, so the check fails and logs the exact reason.

Why Logs Matter Here

Many incidents are not generic "service down" cases. They are specific input failures. The log often tells you which parameter broke the startup path, which saves time and avoids random edits.

How To Investigate

tail -n 20 /var/log/linux-game/reporting-app.log
cat /etc/linux-game/reporting-app.env
/usr/local/bin/reporting-app-check

This sequence lets you see the recent error, inspect the config value behind it, and then rerun the same check after the fix.

Common Mistakes

  • Changing config before reading the error message
  • Editing multiple settings when only one is broken
  • Skipping the final rerun of the startup check

Good Operational Pattern

  1. Read the latest error
  2. Confirm the current failure
  3. Fix the smallest matching config item
  4. Rerun the same check
  5. Verify the new healthy log line

What Success Looks Like

  • The startup check exits successfully
  • The invalid config value is corrected
  • The log now records a healthy startup check
